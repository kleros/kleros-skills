## 🔴 Critical findings

### REV-01: Strip regex matches non-root paths — silent data loss

**Severity:** Critical
**Type:** Correctness bug
**File:** `.github/workflows/sync-master.yml`
**Lines:** 100–101

**Current:**
```bash
git ls-files | grep -E '^(.*_FEEDBACK.*|HANDOVER[^/]*)\.md$' | \
  xargs --no-run-if-empty git rm --quiet
```

**Problem:**
- `^.*_FEEDBACK.*\.md$` matches **any depth**: `kleros-curate/MIGRATION_FEEDBACK.md`, `docs/USER_FEEDBACK_NOTES.md`, anywhere `_FEEDBACK` appears in the path.
- `HANDOVER[^/]*\.md$` is unanchored — matches `kleros-curate/HANDOVER.md` too.
- SEED-002 strip-list says *"root `*_FEEDBACK*.md` / `HANDOVER*.md`"* — the regex is broader than spec.
- A future legitimate `<skill>/MIGRATION_FEEDBACK.md` would silently vanish from every user-installed plugin.

**Fix:**
```bash
git ls-files | grep -E '^[^/]*_FEEDBACK[^/]*\.md$|^HANDOVER[^/]*\.md$' | \
  xargs --no-run-if-empty git rm --quiet
```
The `[^/]*` clauses enforce root-only.

**Status:** addressed in commit (must-fix #1).

### REV-02: Third-party Action handling the App private key is not SHA-pinned

**Severity:** Critical
**Type:** Supply-chain risk
**File:** `.github/workflows/sync-master.yml`
**Line:** 27

**Current:**
```yaml
uses: actions/create-github-app-token@v2
```

**Problem:**
- `@v2` is a mutable tag. The action receives `GH_APP_PRIVATE_KEY` as input.
- If `actions/create-github-app-token` is ever compromised and a malicious `v2` re-tag is pushed (textbook supply-chain attack pattern), the PEM is exfiltrated on the next workflow run.
- Anchore, StepSecurity, and GitHub's own security hardening guide treat unpinned actions handling secrets as the #1 supply-chain attack vector.

**Fix:**
```yaml
uses: actions/create-github-app-token@fee1f7d63c2ff003460e3d139729b119787bc349  # v2.2.2
```
The SHA was resolved live from `gh api repos/actions/create-github-app-token/git/refs/tags/v2` on 2026-06-08; corresponds to v2.2.2 (published 2026-03-13).

**Status:** addressed in commit (must-fix #2). `actions/checkout@v4` and `actions/setup-node@v4` left as version tags for now (they don't see secrets; pinning is best-practice but not as urgent).

### REV-03: Tag push from any commit — privilege amplification

**Severity:** Critical
**Type:** Privilege amplification / access control
**File:** `.github/workflows/sync-master.yml`
**Lines:** 3–7 (trigger)

**Threat:**
- Any contributor with push access to the repo can create a tag pointing to an **arbitrary commit** — including a branch they just pushed with malicious code.
- The workflow runs *at the tagged commit*, runs `npm test` (which may pass for subtle exploits), strips, and force-pushes to master.
- Master is distributed to every user-machine running `marketplace add kleros/kleros-skills` + the Netlify landing page.
- This bypasses any future PR-review gate on `dev` because the attacker never needed dev's review process — they tagged a side branch.

**Fixes (defense in depth):**
1. **GitHub tag protection rule** (repo Settings → Tags → Add rule): restrict creation of `*@v*` and `v*` tags to specific maintainers. Org-admin action.
2. **In-workflow ancestry check** (this commit can implement, repeat once tag protection is in place):
   ```yaml
   - name: Verify tag is on dev ancestry
     run: |
       git fetch origin dev:refs/remotes/origin/dev --depth=50
       git merge-base --is-ancestor ${{ github.sha }} origin/dev \
         || { echo "ERROR: tag commit not reachable from dev — refusing sync"; exit 1; }
   ```
3. **Signed tag verification** (optional, requires team workflow change): `git tag -v ${{ github.ref_name }}`.

**Status:** NOT addressed yet — deferred to "should-fix" follow-up. This is a should-fix in the original red-team output but flagged here as critical because no current mitigation exists. Recommend: add the ancestry check in the should-fix follow-up; the tag protection rule is jaybuidl's manual admin step.

### REV-16: Sanity check shares the strip's blind spots (added 2026-06-11)

**Severity:** Medium
**Type:** Verification gap — defense-in-depth principle
**File:** `.github/workflows/sync-master.yml`
**Lines:** ~163–187 (the "Sanity check stripped tree" step)
**Discovered:** Phase 8 of the deployment plan — `FEEDBACK_FROM_CLAUDE_MARKETPLACE.md` reached master despite the strip step running. REV-09 sanity check did not catch it.

**Problem:**
- The strip step used pattern `^[^/]*_FEEDBACK[^/]*\.md$` (underscore-prefix required).
- The sanity check used pattern `*_FEEDBACK*.md` (same underscore-prefix requirement, in glob form).
- `FEEDBACK_FROM_CLAUDE_MARKETPLACE.md` matched neither (FEEDBACK at position 0, no leading underscore).
- The safety net was blind to exactly the same files the strip was blind to. The strip's miss was not detected.

**Root cause — design principle:**
A sanity check that uses the same predicate as the operation it verifies provides zero defense-in-depth. If the predicate is wrong, both fail silently together. The check exists to detect *deviations* from the expected outcome — it must therefore be expressed in terms of the outcome, not the operation.

**Immediate fix (2026-06-11):** broadened both patterns to drop the underscore-prefix requirement:
- Strip regex: `^[^/]*FEEDBACK[^/]*\.md$|^HANDOVER[^/]*\.md$`
- Sanity check glob: `*FEEDBACK*.md`

This fixes the specific instance but does not address the principle.

**Proper fix (deferred):** the sanity check should assert against a **broader** pattern than the strip, so a stricter strip can never accidentally pass a more permissive verifier:
```bash
# Stronger sanity assertion — catches any root-level upper-case scratch file
if compgen -G "*FEEDBACK*.md" > /dev/null 2>&1 || \
   compgen -G "*_FEEDBACK_*.md" > /dev/null 2>&1 || \
   compgen -G "FEEDBACK*" > /dev/null 2>&1 || \
   compgen -G "HANDOVER*" > /dev/null 2>&1 || \
   compgen -G "SCRATCH*.md" > /dev/null 2>&1 || \
   compgen -G "NOTES*.md" > /dev/null 2>&1; then
  echo "FATAL: root-level scratch-pattern file still present after strip"
  ls -1 *FEEDBACK*.md *_FEEDBACK_*.md FEEDBACK* HANDOVER* SCRATCH*.md NOTES*.md 2>/dev/null
  exit 1
fi
```
Or even simpler — assert that the root file count matches a manually-curated allowlist of known consumer-facing files (SKILL.md, README.md, LICENSE, CHANGELOG.md, robots.txt, sitemap.xml, index.html, plus icons/favicons). Any unexpected root file fails the check.

**Status:** narrow fix applied 2026-06-11 alongside the strip-pattern broadening. Broader principle fix deferred — captured here for the next hardening pass.
