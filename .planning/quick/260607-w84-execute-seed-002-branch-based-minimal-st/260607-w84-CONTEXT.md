## Task Boundary

Execute SEED-002 (`.planning/seeds/SEED-002-exclude-planning-from-plugin-install.md`): set up the branch-based minimal-strip system so end users installing this repo as a Claude Code plugin/marketplace do not receive `.planning/`, dev test/build tooling, or feedback/handover scratch files.

LOCKED design from the seed:
- `dev` branch (or `develop`) holds the full repo — humans work here; PRs target `dev`.
- `master` stays the GitHub default branch but is **derived** from `dev` minus the strip-list, via a GitHub Action. Master is what marketplace-add + plugin-install + Netlify all read.
- Strip-list (remove from master): `.planning/`, `test/`, dev `scripts/`, root `*_FEEDBACK*.md` / `HANDOVER*.md`, build tooling (`package.json`, `yarn.lock`, `.yarnrc.yml`).
- Keep-list on master: `.claude-plugin/`, skill dirs, root `SKILL.md`, `index.html`, `netlify/`, `.well-known/`, `sitemap.xml`, `robots.txt`, favicons, `LICENSE`, `README.md`, `CHANGELOG.md`.

Out of scope for this task: pure-plugin restructure (rejected per seed), separate-repo fallback (rejected per seed), removing the Netlify edge function.

<decisions>
## Implementation Decisions

### Sync trigger (LOCKED)
- The sync Action fires on **tag pushes only**, matching:
  - Skill release tags: `*@v*.*.*` (e.g. `kleros-curate@v1.0.0`, `kleros-ipfs-upload@v1.1.0`)
  - Plain semver tags: `v*` (for whole-plugin or repo-level releases)
- Plus `workflow_dispatch` as a manual override for non-tag emergencies (e.g. landing-page hotfix).
- **Push to `dev` does NOT trigger sync.** Dev is allowed to hold WIP / unstable commits.
- Rationale: existing release convention already uses annotated per-skill tags; tagging IS the "this is releasable" signal. Push-on-dev would ship broken JSON / stale CIDs / unfinished skills to user machines and the Netlify landing page.

### Pre-sync quality gate (LOCKED)
- Before writing master, the Action runs on the dev tree:
  1. `npm test` — must pass
  2. `npm run update-digests` — verify `.well-known/agent-skills/index.json` digests are current; fail if stale
- If either fails, abort sync — master stays at previous good state. Job exits non-zero so the tag push is visibly broken in the Actions tab.
- Rationale: cheap insurance against shipping broken JSON or stale SHA-256 digests to user machines.

### Auth model for the sync push (LOCKED)
- **GitHub App** registered under the `kleros` organization, scoped to `kleros-skills` repo only, `Contents: Read and write` (nothing else).
- Workflow uses `actions/create-github-app-token@v2` to mint a 1-hour installation token; token is auto-revoked at job end.
- Bot identity (`<app-name>[bot]`) appears as the commit actor on master — clearly distinguishable from human pushes.
- Loop-guard: `if: github.actor != '<app-name>[bot]'` on any future jobs that might react to master pushes.
- Rationale: bot identity, bounded leak window (1h vs PAT's up-to-366d), no personal-account coupling, no bus factor, GitHub-recommended pattern for unrestricted automation push. PAT rejected (identity coupling, long-lived secret). PR-based GITHUB_TOKEN rejected (fragile under any future required-checks tightening). Plain GITHUB_TOKEN-direct rejected (cannot bypass branch protection).
- Rejected alternatives documented inline in this conversation; full assessment to be summarised in SUMMARY.md.

### Branch protection rollout (LOCKED — DEFERRED)
- **Phase 1 (this task):** Ship the Action only. Master remains unprotected. Verify a couple of sync cycles work end-to-end before adding restrictions.
- **Phase 2 (follow-up, NOT this task):** Add a **ruleset** (not classic branch protection) on `master`:
  - GitHub App on bypass list with "Always allow" mode.
  - Everyone else (including repo admins) restricted to PR-only on master.
  - Rulesets chosen over classic because classic bypass doesn't cover required status checks.
- Rationale: avoids self-lockout if the Action has bugs in v1. Lower-risk staged rollout. Phase 2 is captured as a follow-up note in SUMMARY.md, not a separate phase yet.

### Untracked root feedback files (LOCKED)
- Current untracked: `CURATE_TOKEN_QUERY_FEEDBACK_POST_REFACTOR.md`, `CURATE_TOKEN_QUERY_FEEDBACK_PRE_REFACTOR.md`, `DNS_AID_FEEDBACK.md`, `FEEDBACK_FROM_CLAUDE_MARKETPLACE.md`, `HANDOVER_FROM_IPFS_GATEWAY_REPO.md`.
- **Commit to `dev` as-is** as part of this task. They match the strip-list pattern (`*_FEEDBACK*.md`, `HANDOVER*.md`) so they will be excluded from master automatically.
- Rationale: preserves content for team reference on dev; user machines never see them; aligned with SEED-002 intent. Avoids the risk that someone later runs `git add .` and accidentally leaks them before the strip is in place.

### Bootstrap path (LOCKED via the trigger decision)
- Create `dev` branch from current `master` HEAD.
- Land all work for this task on `dev` (CONTEXT, PLAN, SUMMARY, GH Action workflow file, README/CLAUDE.md updates, untracked-file commits).
- First master regen happens when the *next release tag* is pushed (the Action only fires on tags). This means master stays as-is until a real release — clean handoff, no premature rewriting.
- For the initial workflow_dispatch sanity check: trigger manually once via `gh workflow run` after the workflow file lands on dev. Inspect the resulting master rewrite. Only after a clean dry-run do we tag the next release.

### dev branch name (LOCKED)
- Use `dev` (not `develop`). Shorter, matches the seed's primary suggestion.
