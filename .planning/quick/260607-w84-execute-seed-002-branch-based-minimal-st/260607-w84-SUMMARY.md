## Preconditions for First Sync Run

The workflow will fail (token error, or hang waiting for Environment approval) until these manual steps are completed:

1. **Register GitHub App in kleros org:**
   - Go to `github.com/organizations/kleros/settings/apps` → "New GitHub App"
   - Name: any (commit identity is now derived dynamically from the App slug — REV-11)
   - Webhook: disabled
   - Permissions: Repository Contents → Read and write; everything else: No access
   - Install on: `kleros/kleros-skills` only
   - After creating: note the App ID, generate a private key (.pem file)

2. **Add repo secrets** (Settings → Secrets and variables → Actions on the repo):
   - `GH_APP_ID` = the numeric App ID (e.g. `123456`)
   - `GH_APP_PRIVATE_KEY` = full PEM content (multiline, including `-----BEGIN RSA PRIVATE KEY-----` header/footer)

3. **Create `production-sync` Environment** (Settings → Environments → New environment) — required by REV-07:
   - Name: `production-sync` (exact match — the workflow references this name)
   - Required reviewers: org owners or named maintainers
   - **Deployment branches AND tags** (critical — both required):
     - Branch: `dev` (allows workflow_dispatch)
     - Tag: `*@v*` (allows skill release tag pushes)
     - Tag: `v[0-9]*` (allows whole-repo release tag pushes)
   - Without the tag patterns, tag-triggered runs are rejected with: *"Tag '...' is not allowed to deploy to production-sync due to environment protection rules"*. **Any new tag pattern added to the workflow's `on.push.tags` list must also be added here.**
   - Without this Environment, the job will hang on first dispatch / never start. Configure before any workflow run.

4. **Tag protection rule** (Settings → Tags → Add rule) — defense in depth for REV-03:
   - Patterns: `*@v*` and `v[0-9]*`
   - Restrict creation to specific maintainers
   - In-workflow ancestry check (REV-03) catches malicious tags after the fact; tag protection prevents them from being pushed in the first place.

5. **Push dev to origin:**
   ```bash
   git push -u origin dev
   ```

Rulesets chosen over classic branch protection because classic bypass doesn't cover required status checks — once required checks are added later, classic bypass would still block the bot. Ruleset bypass is unconditional.

**Precondition for Phase 2:** GitHub App registered + secrets set + at least one clean sync cycle verified.

## Rejected Auth Alternatives

- **PAT (Personal Access Token):** Rejected — identity coupling (tied to a personal GitHub account), long-lived secret (up to 366d), no automatic revocation, bus-factor risk if account owner leaves org.
- **PR-based GITHUB_TOKEN:** Rejected — fragile under any future tightening of required-checks. If required status checks are ever added to master, `GITHUB_TOKEN` cannot bypass them even from Actions, forcing a workflow rewrite.
- **Plain GITHUB_TOKEN direct push:** Rejected — cannot bypass branch protection at all. Once Phase 2 ruleset is in place, `GITHUB_TOKEN` direct pushes would be blocked with no recourse.

Full rationale in CONTEXT.md `## Implementation Decisions → Auth model for the sync push`.

## Deviations from Plan

**1. [Rule 3 - Blocking fix] Agent branch rebased to dev before Task 2**

The worktree-agent branch was created from master HEAD (4e05dc8), which is 3 commits behind dev HEAD (c5b3820). Task 1 files (feedback/handover) were already on dev but not in the worktree. Fast-forwarded agent branch to dev HEAD via `git merge dev --ff-only` before proceeding. This is the correct setup — agent branch now includes Task 1 commits as its base.

**2. [Rule 3 - Fix] force-with-lease bootstrap fix applied**

Per task instructions: added `git fetch origin master:refs/remotes/origin/master || true` step before the push step, so `--force-with-lease` has a known tracking ref. The `|| true` covers the bootstrap case where master doesn't yet exist on origin.

**3. [Rule 3 - Fix] Idempotency check corrected**

PLAN.md's suggested check (`git diff --cached --quiet && git diff --quiet`) would always show changes after `git rm` (deletions are staged). Replaced with counting staged/unstaged file names (`wc -l` on `git diff --cached --name-only`) for accurate detection of the "nothing changed" case.

**4. [Deployment-time discovery] Bootstrap path changed from workflow_dispatch to sentinel tag**

PLAN.md and the deployment plan called for a `workflow_dispatch` dry-run as the first workflow execution. This failed with `HTTP 404: workflow sync-master.yml not found on the default branch`. Root cause: `workflow_dispatch` requires the workflow file to exist on the **default branch** (master), but our master didn't yet have it.

Tag-push events do NOT have this restriction — they use the workflow file at the tagged commit's ref. Bootstrap was performed by pushing sentinel tag `v0.0.0-bootstrap` on dev, which fired the workflow from dev's copy. The workflow stripped and force-pushed master, landing the workflow file on master in the process. From then on, `workflow_dispatch` works normally. The sentinel tag was deleted from origin and locally after the run succeeded.

This discovery is captured in CONTEXT.md `## Post-Execution Learnings` for future maintainers, and the affected Phase 4 (Environment config) step in `## Preconditions for First Sync Run` was updated to require tag patterns in the deployment-tag allowlist.

**5. [Deployment-time discovery] Environment must allow tag patterns explicitly**

Related to deviation #4. The `production-sync` Environment was initially configured per the plan with deployment-branches restricted to `dev` only. The sentinel-tag bootstrap was rejected with: *"Tag 'v0.0.0-bootstrap' is not allowed to deploy to production-sync due to environment protection rules"*.

Fix: added `*@v*` and `v[0-9]*` as tag rules in the Environment's "Deployment branches and tags" section. After the fix, the bootstrap succeeded on re-run.
