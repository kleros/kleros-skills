Content to extract (all from LGTCR §3 unless noted):

Canonical output shape:
  { "columns": [...metaEvidence.metadata.columns verbatim...], "values": { "<col.label>": <userValue>, ... } }
  WHY: Curate UI and subgraph indexers parse this exact envelope; any deviation causes display errors or indexing failures

columns deep-copy rule:
  - Copy columns array verbatim from MetaEvidence.metadata.columns
  - Same objects, same key-value pairs (label, description, type, isIdentifier), same order
  - WHY: the columns array is the schema contract; any edit — even correcting a typo in a description field — means the submitted columns no longer match the on-chain schema that jurors use to evaluate compliance

values construction:
  - One key per col.label; key string must match exactly (spaces, punctuation, case-sensitive)
  - Populate from user input per column type

Output protocol:
  - Print "Fetched columns (verbatim):" and display columns before building values
  - WHY: LLMs tend to paraphrase field names and descriptions; the explicit print step creates an auditable record that makes the verbatim requirement enforceable and gives the user a chance to catch rewriting

Programmatic checklist (5 items from LGTCR §3):
  1. columns deep-equals metaEvidence.metadata.columns (same objects, same order)
  2. keys(values) equals the set of col.label strings across all columns
  3. Each value is non-empty per registry policy (no blank required fields)
  4. Image-type columns: value is a /ipfs/<CID> path, not a gateway URL
  5. Rich-address columns: value is in CAIP-10 format (eip155:<chainId>:0xAddress)

Schema confirmation via NewItem event:
  - Find any NewItem event emitted after the latest MetaEvidence update for this registry
  - Fetch its _data field and decode
  - Confirm columns match (same order, same labels) and observe type encoding (strings vs numbers)
  - WHY: MetaEvidence describes column types in human terms ("address", "text", "image") but not serialization details; past submissions reveal the actual encoding the contract expects

Field-value types and rules:
  - text: plain string
  - longText: plain string (multi-line)
  - link: URL string
  - address: 0x-prefixed Ethereum address
  - richAddress: CAIP-10 format — eip155:<chainId>:0xAddress — ask user for chain if ambiguous
    WHY on CAIP-10: CAIP-10 makes the chain explicit; a bare address on the wrong chain resolves to a different account
  - image: /ipfs/<CID> path — never a gateway URL
    WHY on image path: gateway URLs are mutable and chain-specific; onchain storage must be content-addressed

Placeholder rule:
  - Use PLACE_VALUE_HERE or PLACE_IPFS_URI_HERE as placeholders while drafting
  - Never submit a transaction with placeholder values

Common failure modes (from LGTCR §3):
  - Renaming column labels (e.g. "Name" → "Token Name") — breaks schema contract
  - Reordering columns — contracts may parse positionally
  - Rewriting or inventing column descriptions
  - Changing isIdentifier flags
  - Building the schema from UI assumptions instead of fetching MetaEvidence

  - 4 registry seed templates (Scout §7) — belongs in scout-registries.md
  - Scout mandatory syntax rules restatement — redundant with LGTCR §3 content
