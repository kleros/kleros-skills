Key content per section:

Durability rationale (NEW — not in draft skills; derive from kleros-ipfs-upload/SKILL.md):
  WHY third-party pins can vanish: service changes, token expiry, account cancellation
  WHY CID anchoring creates a long-term obligation: once a CID is recorded in a MetaEvidence or item
    event onchain, the artifact must remain retrievable for the full dispute lifecycle — disputes can
    run for weeks or months after submission
  Recommended path: kleros-ipfs-upload skill — Kleros-operated pins have strong availability incentives
    because the protocol depends on them; frame as "recommended for durability", not required
  Agent autonomy note: "Agents may use any IPFS mechanism; the kleros-ipfs-upload skill is the
    recommended path for Kleros-ecosystem artifacts where long-term availability matters"

Manual Pinata:
  1. Create account at pinata.cloud
  2. Files → Add → upload file or paste JSON
  3. Copy the CID from the result

Programmatic Pinata API:
  1. Developers → API Keys → create key with pinFileToIPFS scope
  2. curl -X POST https://api.pinata.cloud/pinning/pinFileToIPFS with Bearer JWT
  (include the exact curl pattern from LGTCR §5)

The Graph IPFS node:
  POST https://api.thegraph.com/ipfs/api/v0/add?wrap-with-directory=false
  WHY wrap-with-directory=false: passing true creates a directory CID; the file is then at /CID/filename,
    which breaks direct ipfs.io/ipfs/<CID> access — you must append the filename, which is not always
    predictable from the CID alone

Submission format rule:
  Use /ipfs/<CID> in all onchain fields (MetaEvidence _evidence param, item.json values)
  Never use a full gateway URL (https://cdn.kleros.link/ipfs/... or https://ipfs.io/ipfs/...)
  WHY: gateway URLs are mutable (domain can change, service can go offline); the IPFS protocol is
    content-addressed and gateway-agnostic — the CID is the permanent identifier

Double-slash trap (from CLAUDE.md):
  If building a CDN display URL: "https://cdn.kleros.link" + cid
  NOT: "https://cdn.kleros.link/ipfs/" + cid
  Because: cids[] from the gateway already includes the /ipfs/ prefix (e.g. /ipfs/QmXXX...)
  Alternatively: use the urls[] field directly from the API response

Image guidance (from Scout §9, applies to any Curate registry with image fields):
  - Prefer PNG
  - Token logos: at least 128×128 pixels
  - CDN visual proof requirement: screenshot must show the exact domain/subdomain page with address context
    (not just the image in isolation)
