**H1 title:** `# Kleros Curate`

**"Curate in a nutshell" block (~15 lines, per D-09):**
Prose paragraph(s) explaining: what Curate does (decentralised TCR — users submit items backed by a deposit, anyone can challenge, Kleros arbitrates), the deposit/challenge/arbitration cycle, why onchain-first matters. Best source prose: `curate-v1-scout/scout-skills.md` lines 5-19 (per PATTERNS.md). Compress to ~15 lines. Do NOT copy-paste verbatim — write fresh from this framing.

**`## Non-negotiables` section (per D-10):**
Six cross-flavor safety rules as a structured list (no markdown tables — D-12). Use bold emphasis on the key phrase per each rule. Exact rules from PATTERNS.md "Non-negotiables block":
1. Never guess / invent / approximate amounts, addresses, schemas, or parameters.
2. Onchain state + onchain logs are the source of truth for deposits, arbitration cost, challenge deposits, appeal status, and MetaEvidence URI.
3. Never assume a "standard token schema" — only the current MetaEvidence is authoritative.
4. Never rewrite the schema: `item.json.columns` must be copied verbatim from MetaEvidence; only `values` is dynamic.
5. Never include "typical ranges" / estimates for deposits or fees — only report live-read values.
6. `eth_getCode` before declaring any address is or isn't a contract.

**`## Which Curate flavor are you using?` section (per D-01 through D-05):**
Three-step routing tree as a structured list. No markdown tables. Exact structure from PATTERNS.md "Routing decision tree pattern":

- LGTCR triggers (default): "Curate", "LGTCR", "LightGeneralizedTCR", "Light Curate", registry, "addItem" → Read `references/light-curate.md`

Step 2 — Ambiguous ("Curate" with no flavor hint, per D-03):
- Interactive session: ask one question (1-sentence descriptions of each of the 3 flavors)
- One-shot/non-interactive: default Light Curate, then progressively correct: if address matches one of 4 Scout Gnosis registry addresses → load Scout overlay; if contract introspection reveals PGTCR hallmarks → pivot to `references/stake-curate.md`

Step 3 — Contract-type verification (if address provided, per D-05):
- SKILL.md says "See the flavor reference file for hallmark calls" — do NOT embed function signatures here (belongs in reference files)

**`## Action index` section (per D-11, D-12):**
Structured list mapping user intents to reference files. No markdown tables. Bold label → arrow → file path. Grep patterns inline for large files only. Use exact format from PATTERNS.md "Action index pattern":

- Fund an appeal → flavor reference file
